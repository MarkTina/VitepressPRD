# 3. 用户画像

## 用户角色

### 角色 1：产品经理（主要用户）

| 属性 | 描述 |
|------|------|
| 角色名称 | 产品经理 |
| 年龄/职位 | 25-35 岁，初级到资深 PM |
| 使用场景 | 撰写 PRD、管理需求版本、与研发沟通 |
| 核心痛点 | 文档撰写耗时、结构不统一、版本难追溯 |
| 期望价值 | 快速产出高质量 PRD，聚焦需求挖掘 |

**典型使用场景：**

1. 接到新需求，想快速搭建 PRD 框架
2. 需求变更，需要更新 PRD 并保留历史版本
3. 与研发评审需求，需要清晰的流程图和验收标准
4. 产品上线后，需要基于 PRD 生成用户文档

### 角色 2：研发工程师

| 属性 | 描述 |
|------|------|
| 角色名称 | 研发工程师 |
| 年龄/职位 | 23-32 岁，前后端开发 |
| 使用场景 | 阅读 PRD 理解需求、技术方案设计 |
| 核心痛点 | PRD 结构混乱、信息分散、缺少验收标准 |
| 期望价值 | 结构清晰、信息完整、可快速定位关注点 |

**典型使用场景：**

1. 接到开发任务，查阅 PRD 了解功能详情
2. 需要了解业务流程，查看 Mermaid 流程图
3. 需要了解接口定义，查看 API 文档章节
4. 需要了解验收标准，对照 checklist 自测

### 角色 3：测试工程师

| 属性 | 描述 |
|------|------|
| 角色名称 | 测试工程师 |
| 年龄/职位 | 24-30 岁，QA |
| 使用场景 | 编写测试用例、验收功能 |
| 核心痛点 | 需求描述模糊、缺少边界情况、验收标准不明确 |
| 期望价值 | 需求清晰可测、边界情况完整、验收标准具体 |

## 使用场景

### 场景 1：快速创建新需求 PRD

**触发条件：** 产品经理接到新功能需求
**用户行为：**
1. 运行 `npm run new-version v1.1.0` 创建新版本
2. 与 AI 讨论需求，梳理清楚功能点
3. AI 按模板生成完整 PRD
4. 微调后完成 PRD

**期望结果：** 30 分钟内产出结构完整的 PRD 初稿

### 场景 2：查看历史需求

**触发条件：** 研发需要了解之前版本的功能逻辑
**用户行为：**
1. 打开 PRD 文档站点
2. 通过 Navbar 版本选择器切换到目标版本
3. 在 Sidebar 中定位到对应功能模块

**期望结果：** 快速找到历史版本的需求描述

### 场景 3：需求评审

**触发条件：** 产品组织需求评审会
**用户行为：**
1. 打开 PRD 文档站点
2. 按章节逐一讲解需求
3. 研发提问时快速定位到对应内容

**期望结果：** 评审过程高效，信息查找迅速
