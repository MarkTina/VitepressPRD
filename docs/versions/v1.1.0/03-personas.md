# 3. 用户画像

## 用户角色

### 角色 1：平台运营管理员

| 属性 | 描述 |
|------|------|
| 角色名称 | 平台运营管理员 |
| 年龄/职位 | 28-40 岁，运营经理 / 后台管理员 |
| 使用场景 | 管理短信模板与配额；配置网点收款模式；查看发送/支付数据 |
| 核心痛点 | 短信渠道切换需要研发介入；网点收款配置每次都要研发改代码 |
| 期望价值 | 可视化后台自主完成配置，无需研发介入 |

**典型使用场景：**

1. 新开一家洗衣网点，在后台创建支付配置并绑定到该网点
2. 双十一期间调整短信日配额，防止超发
3. 查看某网点近 30 天支付汇总，核对分账金额
4. 审核新创建的短信模板，通过后启用

### 角色 2：加盟商（洗衣网点老板）

| 属性 | 描述 |
|------|------|
| 角色名称 | 加盟商 |
| 年龄/职位 | 30-50 岁，个体经营者 |
| 使用场景 | 查看网点收款情况；确认分账明细 |
| 核心痛点 | 不清楚每笔订单平台抽了多少、自己实收多少 |
| 期望价值 | 手机端可查看每日收款和分账明细，资金透明 |

**典型使用场景：**

1. 每天打烊后查看当日收款总额和分账后实际到账金额
2. 收到分账异常通知后联系平台核实
3. 新网点开业前，由平台协助绑定收款配置

### 角色 3：后端研发工程师

| 属性 | 描述 |
|------|------|
| 角色名称 | 后端研发工程师 |
| 年龄/职位 | 24-35 岁，Java/Go 开发 |
| 使用场景 | 接入短信微服务；对接支付配置系统；处理支付回调 |
| 核心痛点 | 短信对接无统一文档；支付变更每次都要改动核心代码 |
| 期望价值 | 短信服务接入简单（读文档 + API Key 即可）；支付配置变更不影响业务代码 |

**典型使用场景：**

1. 新业务线接入短信服务：申请 API Key → 调用 send 接口 → 完成
2. 新增支付模式（如云闪付）：实现支付接口 → 注册到支付配置中心 → 网点即可选用
3. 排查短信发送失败：在短信服务后台查看发送日志和错误原因

### 角色 4：财务人员

| 属性 | 描述 |
|------|------|
| 角色名称 | 财务专员 |
| 年龄/职位 | 26-40 岁，财务会计 |
| 使用场景 | 日终对账；分账结算审核；异常交易处理 |
| 核心痛点 | 多网点多渠道对账需手工汇总 Excel，耗时且易出错 |
| 期望价值 | 自动对账，差异自动标记，一键导出报表 |

**典型使用场景：**

1. 每月 5 号导出上月各网点支付汇总用于结算
2. 收到对账异常告警后查看具体交易明细
3. 核对微信分账金额是否与预设规则一致

## 使用场景

### 场景 1：新网点开业 - 配置收款

**触发条件：** 一家新的自助洗衣加盟店即将开业
**用户行为：**
1. 运营管理员在后台创建支付配置（选择微信服务商分账模式，填写商户号、分账规则）
2. 在网点管理页将新配置绑定到该网点
3. 网点设备端发起一笔测试支付，确认收款成功
**期望结果：** 10 分钟内完成配置，网点开业即可正常收款

### 场景 2：短信渠道切换

**触发条件：** 公司决定将短信渠道从阿里云切换到腾讯云
**用户行为：**
1. 研发在短信微服务中新增腾讯云渠道适配器
2. 运营管理员在后台将短信模板的渠道类型切换为腾讯云
3. 观察发送成功率，确认正常后停用阿里云渠道
**期望结果：** 业务方无感知，快递/洗衣等业务短信发送不受影响

### 场景 3：支付分账异常处理

**触发条件：** 某笔订单微信分账失败（接收方账户异常）
**用户行为：**
1. 系统自动记录异常，触发告警
2. 财务人员查看分账失败详情，联系加盟商核实账户信息
3. 修正后手动发起重试分账
**期望结果：** 异常分账不阻塞主交易，事后可手动补分账
